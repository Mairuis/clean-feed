# Clean Feed UI Requirements

## 1. Product Positioning

Clean Feed is a browser extension for cleaning video feeds across YouTube and Bilibili.

Core promise:

> 还我一个干净的信息流。

The product should feel simple, quiet, and restorative. It is not a power-user rule editor. The user should describe what they dislike in natural language, let AI generate the filtering configuration, and only review the result when needed.

## 2. Target UX

The UI should be minimal and confidence-building:

- The default experience is "describe once, then let it run".
- Complex settings are hidden behind secondary menus or dialogs.
- The main screen should not look like a dashboard full of cards.
- Avoid exposing low-level rule editing as the primary interaction.
- Use natural-language configuration as the main user action.
- Keep the visual mood clean, green, fresh, and calm.

Avoid:

- Red or warning-heavy visual themes.
- Dark, heavy, or oppressive palettes.
- Dense card stacks.
- Manual rule tables as the main UI.
- Asking users to repeatedly choose automation settings that should be automatic.

## 3. Core Product Model

Clean Feed has two concurrent filtering layers.

### 3.1 Fast Programmatic Filtering

This layer runs immediately and solves the latency problem before LLM responses arrive.

It currently supports:

- Keyword filtering.
- Video duration filtering.
- Platform-specific short video filtering, such as YouTube Shorts and Bilibili short-form video content.

These rules are generated by AI from the user's natural-language preference. The user can review them, but v1 should not encourage manual editing.

### 3.2 LLM Post-Review Filtering

This layer runs after the first pass. The page may render normally first, then Clean Feed sends video title, channel, URL, and duration to the background worker. The LLM returns a quality judgment.

Verdicts:

- `low_quality`: blur the video card.
- `high_quality`: keep the video.
- `uncertain`: keep the video.

The UI must set the expectation that AI review is a second pass, not an instant hard block.

## 4. Content Filtering Behavior

Filtered videos must not disappear by default. The whole video card should be visually blurred.

Requirements:

- Apply a Gaussian blur to the whole card.
- Reduce visual salience, but preserve layout so the feed does not jump.
- On mouse hover, remove the blur so the user can inspect the original card.
- On keyboard focus within the card, remove the blur for accessibility.
- The blur interaction must feel lightweight and reversible.

Optional design idea:

- A small subtle reason label can appear on the blurred card, such as "短视频", "关键词命中", or "AI 判断低质". It must not clutter the feed.

## 5. Platform Feedback

Clean Feed can automatically click "Not interested" or "Dislike" on filtered items so recommendation systems learn from the user's preference.

Product decision for v1:

- This should be automatic by default.
- Do not make this a prominent manual setting on the main screen.
- If surfaced, place it in an advanced or secondary area.
- The UI copy should explain this as platform feedback, not as a destructive action.

The UI should not ask the user to manually click per-video feedback controls.

## 6. Main Surfaces

### 6.1 Popup

Purpose:

- Quick status and emergency control.

Expected contents:

- Product name: Clean Feed.
- Short tagline: 还我一个干净的信息流.
- Master enable/disable switch.
- Compact status for fast rules.
- Compact status for AI post-review.
- One button to open settings.

Do not include:

- AI API form fields.
- Rule editing.
- Long explanations.
- Multiple configuration panels.

Suggested size:

- Around 320-380 px wide.
- Designed for quick glance and one-click control.

### 6.2 Settings Page

Purpose:

- The main configuration experience.

Primary interaction:

- A single natural-language composer where the user describes what they want to filter.
- A primary action such as "生成净化方案".
- A concise status line showing whether the plan was generated, saved, or failed.

Above-the-fold contents:

- Brand/header.
- Natural-language text area.
- Primary generate/apply button.
- Small status indicators:
  - Fast rules active count.
  - AI review status.
  - Platform feedback status.
- Secondary entry for AI connection.

Secondary contents:

- "查看当前方案" or similar collapsible review section.
- Show generated quick rules and LLM review strategy in a readable, non-edit-heavy way.
- The generated plan should feel like an AI-produced summary, not a spreadsheet.

Do not make:

- A rule builder as the main view.
- A pile of separate cards for every setting.
- A technical dashboard.

### 6.3 AI Connection Dialog

Purpose:

- Configure provider connection only when needed.

Fields:

- API Base.
- API Key.
- Model.
- Test connection.
- Save.

Defaults:

- API Base: `https://openrouter.ai/api/v1`
- Model: `anthropic/claude-haiku-4-5`

Design guidance:

- This belongs in a secondary dialog, sheet, or popover.
- It should not dominate the main settings page.
- API Key should be clearly private and stored locally.
- Failure states should be calm and actionable.

### 6.4 In-Page Feed Treatment

Purpose:

- Make filtered content less attention-grabbing without hiding context.

Visual behavior:

- Blurred cards remain in their original feed position.
- Hover/focus reveals original content.
- No layout shift.
- No huge overlays.
- No aggressive warning colors.

The in-page treatment should feel like "visual muting", not deletion.

## 7. First-Run Flow

Ideal flow:

1. User installs the extension.
2. Popup shows Clean Feed is enabled and invites opening settings.
3. Settings page asks for a natural-language filtering preference.
4. If AI is not connected, the UI prompts the user to connect AI through the secondary connection dialog.
5. User saves API Base, API Key, and Model.
6. User clicks "生成净化方案".
7. Clean Feed shows a compact generated summary and starts applying filters.

The user should never need to understand the internal storage model, background worker, or rule engine.

## 8. Returning User Flow

Expected flow:

1. User opens settings.
2. Existing preference text is visible.
3. Current generated plan is available in a collapsed review section.
4. User can rewrite the preference and regenerate.
5. The extension applies the new generated plan.

Editing should happen by rewriting natural language, not by directly editing many individual rules.

## 9. Error and Empty States

Important states:

- AI key missing.
- API Base permission not granted.
- Test connection failed.
- AI generation failed.
- AI disabled or unavailable.
- No generated rules yet.

Tone:

- Clear, calm, concise.
- No alarming red UI unless the action is genuinely destructive.
- Error messages should explain what to do next.

Examples:

- "需要先连接 AI，才能生成净化方案。"
- "连接失败，请检查 API Key、API Base 和模型名称。"
- "AI 暂不可用，快速规则仍会继续运行。"

## 10. Visual Direction

Preferred mood:

- Fresh.
- Clean.
- Quiet.
- Light.
- Focused.

Preferred palette:

- Soft green as the main brand color.
- Teal or blue-green accents.
- Neutral off-white or very light gray backgrounds.
- Dark neutral text for readability.

Avoid:

- Red as a primary color.
- Heavy black backgrounds.
- Purple-blue gradient SaaS look.
- Brown, orange, or beige-heavy palettes.
- Decorative blobs or busy ornamental backgrounds.

UI shape:

- Compact controls.
- Moderate border radius, around 8 px or less.
- Clear spacing.
- Avoid nested cards.
- Use cards only when grouping repeated generated rule items genuinely helps.

Typography:

- Chinese-first readability.
- Clear hierarchy.
- No oversized marketing hero treatment.
- No long instructional paragraphs in the main interface.

## 11. Component Requirements

Potential components:

- Master switch.
- Status pill or compact metric.
- Natural-language text area.
- Primary action button.
- Secondary icon/text button for AI connection.
- Dialog/sheet for AI connection.
- Collapsible generated-plan review.
- Small generated rule rows.
- Subtle in-page blur treatment.

Button hierarchy:

- Primary: generate/apply plan.
- Secondary: AI connection, test connection, open settings.
- Destructive actions are not part of v1 main UI.

## 12. Accessibility

Requirements:

- All controls keyboard accessible.
- Dialog focus should be trapped while open.
- Blurred video cards reveal on `:focus-within`.
- Text contrast must be readable.
- Status updates should be available to screen readers where practical.
- Buttons and inputs need clear labels.

## 13. Responsive Behavior

Popup:

- Fixed compact width.
- No horizontal scrolling.
- Important status visible without scrolling.

Settings page:

- Comfortable on desktop extension options window.
- Still usable in a narrow browser extension settings pane.
- Main composer should remain the focus.
- Dialog fields should stack on small widths.

In-page treatment:

- Must not break YouTube or Bilibili layouts.
- Must preserve card dimensions.

## 14. Non-Goals for v1 UI

Do not design these as first-class v1 experiences:

- Full manual rule editor.
- Analytics dashboard.
- Per-site advanced tuning panels.
- Account system.
- Cloud sync UI.
- Mobile web-specific interface.
- Per-video manual feedback controls.
- Complex onboarding carousel.

## 15. Design Deliverables Requested

Please produce:

- Popup design.
- Settings page design.
- AI connection dialog design.
- Generated plan review design.
- In-page blurred card treatment.
- Empty, loading, success, and error states.
- Color tokens and typography recommendations.

The design should make Clean Feed feel like a calm AI assistant that quietly cleans the feed, not a settings-heavy blocker extension.
